# アプリケーション コードレビュー報告書（UI応答遅延・クラッシュ原因解析）

本ドキュメントは、現在のハイブリッド構成（Flutter + Jetpack Compose/Androidネイティブ処理）におけるパフォーマンス低下、UI応答遅延、およびクラッシュ・バグの根本原因を調査・分析した結果をまとめたものです。ソースコードの変更は行わず、解析結果のみを記載しています。

---

## 1. UIの応答遅延・カクつき（Stuttering/Jank）の原因

Flutter UIとAndroid (Kotlin) 間の状態同期処理プロセスに、極めて重いボトルネックが存在しています。

### A. 全状態の過剰な同期とシリアライズ (EventChannelの乱用)
`PaintMethodChannelHandler.kt` 内の `startStateObserver()` で、**すべての状態Flow** (ブラシ設定、カラーヒストリー、そして**全レイヤー情報**) がマージされ、何らかの状態が変化するたびにFlutterへ同期されています。
- ブラシのサイズスライダーを動かしたり、キャンバスにストロークを描くたびに、**すべてのレイヤースタックやカラーデータを含む巨大なマップ (`buildStateMap()`) が再構築**され、JNIを経由してFlutterへ送信されています。
- これにより、大量のオブジェクトアロケーション（String キャストや Map 作成）が毎フレーム発生し、**ガベージコレクション (GC) が頻発してUIスレッドを著しくブロック**しています。

### B. Flow の `debounce(16L)` の仕様によるフリーズ現象
上記の状態同期において、負荷を減らす意図で `.debounce(16L)` が適用されています。しかし、Kotlin Coroutines の `debounce` は「イベントが**静止してから**指定時間経過後に発行」する仕様です。
- ユーザーがスライダーを引き続けたり、画面にペンを走らせて連続的にイベントが発生している間は、UIへの状態送信がパタリと止まります。
- 指を離して初めてイベントの送信が再開されるため、「**操作中だけUIが固まり、操作をやめると一気に反映される**」という極めて応答性の悪い状態を引き起こしています。

## 2. クラッシュ（強制終了・ANR）の主な原因

### A. ライフサイクルイベント時のUIスレッドブロック (ANR)
`PaintFlutterActivity.kt` において、アプリがバックグラウンドに移行するタイミングでプロジェクトファイルの保存が行われていますが、実装に致命的な問題があります。
- `onPause()` および `onStop()` の両方で `viewModel.saveCurrentProject()` が **同期的に（Main/UIスレッド上で）呼び出されています。**
- 画像レイヤーなどの保存・圧縮といった非常に重いファイルI/O処理がUIスレッドを完全にロックするため、OSの「応答なし (ANR: Application Not Responding)」検知によってアプリがクラッシュします。
- さらに、`onPause` と `onStop` で2回連続でセーブ処理が走るため、書き込みの競合やファイル破損のリスクも高くなっています。

### B. Activity再生成に伴う状態喪失（サイレント・クラッシュ）
画像出力/読み込み機能において、Android標準のファイルピッカーを利用していますが、Activity Result待ちの状態管理がセーフではありません。
- `PaintFlutterActivity` 内の `pendingExportFormat` 変数（インポート/エクスポートするフォーマットを記憶する変数）が揮発性の通常のクラス変数として定義されています。
- 他のメモリ喰いのアプリを開いている間にAndroid OSが当該Activityを破棄・再生成した場合、このフラグが `null` にリセットされます。
- ピッカーから戻った際、`pendingExportFormat` が失われているため、`onActivityResult` でエクスポート/インポート処理が中断するか、予期せぬ挙動（何も起きないバグ）につながります。

### C. レイヤー操作の同期的実行
`duplicateLayer` や `batchMergeLayers` など一部の重い処理は `launchHeavy` (バックグラウンドスレッド) で呼ばれていますが、`moveLayerUp` や `setLayerOpacity` などの操作はMethodChannel上で直接（UIスレッドで）実行されています。
レイヤーの再描画やGLテクスチャの切り替えを伴う場合、これらもUI応答を止める要因やクラッシュの原因になり得ます。

## 3. その他のバグ・アーキテクチャ上の懸念点

### A. 自動保存と手動保存の競合（Race Condition）
`PaintViewModel.kt` の `startAutoSave()` にて60秒間隔の自動保存 (`saveCurrentProject()`) が非同期実行されていますが、ユーザーが「ギャラリーに戻る」ボタンを押した際（または `onPause` が呼ばれた際）にも同メソッドが呼ばれます。
- `saveCurrentProject` の関数内でファイルへの排他制御（Mutex等）が行われていない場合、自動保存の書き込み中とギャラリーに戻る際の書き込みが同時に発生し、プロジェクトファイル(`.ppaint`)が破損する危険性があります。

### B. PlatformView のメモリ管理
`PaintCanvasPlatformView` において、`GLSurfaceView` をFlutterのView階層に埋め込んでいますが、`dispose()` メソッドの中身が空（`// GLSurfaceView のライフサイクルは Activity に委譲`）になっています。
Flutter側から不要になって破棄された場合でも、ネイティブ側のビューやGLのコンテキストが確実に解放されるか保証されていないため、長時間のアプリ使用や画面切り替えでメモリリークを引き起こす恐れがあります。

---

## 総評
1. 現在の**「ネイティブとFlutterの間で、変更箇所だけでなく常に全状態をデバウンス同期する」**という設計が、UIの応答遅延の全ての根源です。状態同期対象を細分化し、必要なイベントのみ流すアーキテクチャへ見直す必要があります。
2. クラッシュ報告の大半は、バックグラウンド遷移時（`onPause/onStop`）の**メインスレッドでのファイル保存**に起因していると推測されます。I/O処理の非同期化が早急に必要です。
